Dify can export structured trace data to external observability platforms. Tracing gives you a detailed, per-request view of LLM inputs and outputs, latency, token counts, tool calls, and retrieval steps—beyond what the built-in monitoring dashboard provides.

## What is traced

For every application request, Dify sends the following data to the configured tracing provider:

| Trace data | Description |
|------------|-------------|
| Conversation / message ID | Dify-internal identifiers for correlating traces back to logs. |
| LLM inputs | The full prompt sent to the model, including system prompt and context. |
| LLM outputs | The model's complete response. |
| Token counts | Input tokens, output tokens, and total tokens per LLM call. |
| Latency | Time from request start to first token, and total generation time. |
| Tool calls | Name, inputs, and outputs of any tool invoked during an agent run. |
| Dataset retrieval | Documents retrieved from knowledge bases, including scores. |
| Moderation | Whether the moderation check flagged the message and the action taken. |
| Workflow steps | Per-node trace data for workflow applications, including inputs/outputs of each node. |
| Errors | Error messages and stack traces when a request fails. |

## Enabling tracing

<Steps>
  <Step title="Open application settings">
    From the Dify dashboard, open your application and click **Settings** in the left navigation.
  </Step>
  <Step title="Go to Monitoring">
    Click the **Monitoring** tab within application settings.
  </Step>
  <Step title="Configure a provider">
    Click **Configure** next to the provider you want to enable. Enter the required credentials (see provider-specific sections below).
  </Step>
  <Step title="Enable tracing">
    Toggle the provider to **Enabled** and click **Save**. Traces will be sent for all subsequent requests.
  </Step>
</Steps>

<Warning>
Only one tracing provider can be active per application at a time. Enabling a new provider disables the previous one.
</Warning>

## How tracing works internally

When tracing is enabled for an application, Dify enqueues trace tasks asynchronously via a `TraceQueueManager`. Tasks are batched (default batch size: 100, flush interval: 5 seconds) and dispatched to Celery workers, which serialize the trace data and send it to the configured provider. This means tracing has minimal impact on request latency.

Trace data is serialized per event type:

- `MESSAGE_TRACE` — a complete chat message round-trip
- `WORKFLOW_TRACE` — a workflow execution with all node results
- `TOOL_TRACE` — a single agent tool invocation
- `DATASET_RETRIEVAL_TRACE` — a knowledge base retrieval step
- `MODERATION_TRACE` — a content moderation check
- `SUGGESTED_QUESTION_TRACE` — suggested follow-up question generation
- `GENERATE_NAME_TRACE` — conversation title generation
