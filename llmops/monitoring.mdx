title: Monitoring and metrics
description: Track usage, performance, and cost for your Dify applications with built-in dashboards.
---

The **Overview** dashboard in each Dify application provides time-series metrics covering usage, performance, and cost. Use it to understand traffic patterns, identify latency spikes, and track token spend over time.

## Accessing the dashboard

1. Open your application from the Dify dashboard.
2. Click **Monitor** in the left navigation.
3. The **Overview** tab is selected by default and shows the metrics dashboard.

## Available metrics

### Usage metrics

| Metric | Description |
|--------|-------------|
| Total messages | The total number of messages processed in the selected period. |
| Active users | Unique end users who sent at least one message. |
| Conversations | Total distinct conversation sessions. |
| Messages per conversation | Average number of turns per session. |

### Performance metrics

| Metric | Description |
|--------|-------------|
| Average latency | Mean time (seconds) from user message receipt to first token returned. |
| Token throughput | Tokens generated per second, averaged across all requests. |

### Token and cost metrics

| Metric | Description |
|--------|-------------|
| Total input tokens | Tokens sent to the model (prompt + context) across all messages. |
| Total output tokens | Tokens generated by the model across all messages. |
| Total tokens | Combined input and output token count. |

<Info>
Dify tracks token usage per message and stores it in the conversation log. Token cost in dollars is not calculated by Dify directly—you can derive cost by multiplying token counts by your model provider's per-token pricing.
</Info>

## Time range selection

All metrics support the following time windows:

- Last 24 hours
- Last 7 days
- Last 30 days
- Custom date range

Metrics are plotted as time-series charts so you can spot trends and anomalies.

## Model configuration tracking

Dify records the model configuration (model name, temperature, max tokens, prompt template version) active at the time each message was processed. When you browse a log entry, you can see exactly which configuration produced that response. This makes it possible to correlate prompt or model changes with changes in quality metrics.

## Evaluation with historical data

You can use logged conversations as an evaluation dataset:

1. Export logs as a CSV from **Logs & Ann.**
2. Use the CSV to run batch evaluations against prompt or model changes.
3. Compare annotation-based expected answers against new model responses.

For automated evaluation pipelines, consider connecting a tracing integration such as Langfuse or Arize Phoenix, which provide dedicated evaluation tooling.

## Grafana integration

For self-hosted deployments that require richer dashboards, a community-contributed Grafana dashboard is available. It uses Dify's PostgreSQL database as a data source and provides metrics at the granularity of apps, tenants, messages, and more.

- [Grafana Dashboard by @bowenliang123](https://github.com/bowenliang123/dify-grafana-dashboard)

<Tip>
The Grafana dashboard is maintained by the community. It requires direct access to the Dify PostgreSQL database.
</Tip>
