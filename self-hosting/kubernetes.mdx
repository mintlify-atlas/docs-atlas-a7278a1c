Kubernetes is recommended for production deployments that require high availability, horizontal scaling, or integration with existing cloud-native infrastructure.

## Basic deployment steps

<Steps>
  <Step title="Add the Helm chart repository">
    Steps vary by chart. For example, with the douban chart:

    ```bash
    helm repo add douban https://douban.github.io/charts/
    helm repo update
    ```
  </Step>
  <Step title="Create a values file">
    Copy the chart's default `values.yaml` and customize it:

    ```bash
    helm show values douban/dify > my-values.yaml
    ```

    At minimum, configure:
    - Database connection (host, credentials)
    - Redis connection
    - Storage backend
    - Ingress hostname
    - `SECRET_KEY`
  </Step>
  <Step title="Create Kubernetes secrets">
    Store sensitive values as Kubernetes Secrets rather than in `values.yaml`:

    ```bash
    kubectl create secret generic dify-secrets \
      --from-literal=SECRET_KEY=$(openssl rand -base64 42) \
      --from-literal=DB_PASSWORD=your-db-password \
      --from-literal=REDIS_PASSWORD=your-redis-password
    ```

    Reference these secrets in your `values.yaml` using `secretKeyRef`.
  </Step>
  <Step title="Install the chart">
    ```bash
    helm install dify douban/dify \
      --namespace dify \
      --create-namespace \
      --values my-values.yaml
    ```
  </Step>
  <Step title="Verify the deployment">
    ```bash
    kubectl get pods -n dify
    kubectl get svc -n dify
    ```

    Wait until all pods reach `Running` status.
  </Step>
</Steps>

## Configuring secrets

Never store sensitive values in Helm values files committed to version control. Use Kubernetes Secrets or a secrets management solution:

```yaml
# Example: referencing a secret in a values.yaml
api:
  env:
    - name: SECRET_KEY
      valueFrom:
        secretKeyRef:
          name: dify-secrets
          key: SECRET_KEY
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: dify-secrets
          key: DB_PASSWORD
```

For production, consider using [External Secrets Operator](https://external-secrets.io/) or [Vault](https://www.vaultproject.io/) to inject secrets from your secrets management system.

## Ingress configuration

Dify requires two ingress routes: one for the API/backend and one for the frontend. An example Nginx ingress configuration:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: dify-ingress
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "3600"
spec:
  ingressClassName: nginx
  rules:
    - host: console.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: dify-web
                port:
                  number: 3000
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: dify-api
                port:
                  number: 5001
          - path: /console/api
            pathType: Prefix
            backend:
              service:
                name: dify-api
                port:
                  number: 5001
```

<Note>
Dify SSE (Server-Sent Events) streams require long-lived HTTP connections. Set ingress proxy timeouts to at least 3600 seconds.
</Note>

## Persistent volume claims

The following services require persistent storage:

| Service | Purpose | Recommended storage class |
|---------|---------|---------------------------|
| PostgreSQL | Primary database | SSD-backed block storage |
| Redis | Cache and queue | SSD-backed block storage |
| API/Worker | Uploaded files (if using local storage) | ReadWriteMany NFS or object storage |
| Vector store | Embedding data | SSD-backed block storage |

For the `api` and `worker` services to share the same file storage volume, use a `ReadWriteMany` storage class or configure an external object storage backend (S3, Azure Blob, GCS) instead.

## Horizontal scaling

| Component | Scalable | Notes |
|-----------|----------|-------|
| `api` | Yes | Stateless. Scale horizontally behind a load balancer. |
| `worker` | Yes | Stateless Celery workers. Scale based on queue depth. |
| `web` | Yes | Stateless Next.js. Scale horizontally. |
| PostgreSQL | Limited | Use a managed PostgreSQL service (RDS, Cloud SQL) for HA. |
| Redis | Limited | Use a managed Redis service (ElastiCache, Cloud Memorystore) for HA. |
| Vector store | Depends | Weaviate and Qdrant support clustering; pgvector scales with PostgreSQL. |

```bash
# Scale the API deployment
kubectl scale deployment dify-api --replicas=3 -n dify

# Scale workers
kubectl scale deployment dify-worker --replicas=5 -n dify
```
